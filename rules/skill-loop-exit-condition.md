# 技能循环必须有退出条件

创建或修改任何技能（Skill）时，必须确保文档中所有循环/重复结构都有显式的退出条件。

## 规则内容

你应当在技能文档中检查以下结构，并确保每个都有明确退出条件：

### 需要检查的循环模式

| 模式 | 典型场景 | 退出条件示例 |
|------|----------|-------------|
| 编辑-确认循环 | 用户反复修改标题/内容/commit message | `最多 3 轮，第 3 轮后直接询问 Y/N` |
| 补充信息-重新分析循环 | 用户多次补充信息要求重新分类 | `最多 2 轮，第 2 轮后直接给出推荐` |
| 修复-验证循环 | 修复问题后再次检查 | `无 🔴 问题时退出` |
| 等待-重试循环 | 等待用户输入后重试某操作 | `最多 N 次重试，之后终止` |

### 退出条件必须满足

- **具体**：给出明确的数字上限或结束条件，不能是「用户满意时退出」这类主观表述
- **可执行**：AI 在循环达到上限时知道该做什么（如「直接使用当前结果」「终止流程」「降级处理」）
- **有兜底**：除了正常退出路径，还需有「用户中途取消 → 立即终止」的全局出口

## 适用范围

- 创建新技能时自动应用
- 审计技能时作为 D4 维度的判定依据
- 修改技能时检查是否引入了新循环但遗漏退出条件

## 反例（错误示范）

```
用户 A 或 B 都不满意 → 重新生成
用户提供修改意见 → 按意见调整后再次确认
```

上方写法缺少退出上限，AI 会陷入无限循环。

## 优先级

高。此前审计的 git-commit、session-summary、idea-classifier 三个技能均在 D4 维度出现相同缺陷，修复成本低但遗漏频率高，必须在创建阶段就杜绝。

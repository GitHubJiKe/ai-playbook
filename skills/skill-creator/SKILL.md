# Skill Creator — 资产创建器

帮用户创建高质量 AI Playbook 资产。内置从三次审计中提炼的全部质量规则，在生成阶段自动杜绝常见缺陷。

> ⚠️ **核心原则**：质量前移，不做事后审计。生成时就满足六维度标准，而非生成后再修修补补。对技能类型执行完整六维度检查，对其他类型按需降级。

---

## Step 1：明确创建目标

### 1.1 如果用户已通过 idea-classifier 确定了类型

直接使用已确认的类型和名称，跳到 Step 2。

### 1.2 如果用户直接描述需求

询问以下信息（合并为一次提问）：

```
好的，要创建一个新资产。请告诉我三件事：

1. 这个资产叫什么名字？（简洁的名称即可）
2. 它解决什么问题？（一句话描述核心功能）
3. 你知道它属于哪种类型吗？（可选：命令/提示词/角色/规则/脚本/技能/工具/文章。不确定的话我帮你判断）
```

### 1.3 快速分类（如果用户不确定类型）

内部对用户描述做快速分类，参考标准：

| 特征 | 类型 |
|------|------|
| 一句话单轮即完成 | `commands/` |
| 有占位变量的可复用模板 | `prompts/` |
| 定义 AI 人设+行为边界 | `roles/` |
| 跨场景行为约束 | `rules/` |
| 需写代码自动化运行 | `scripts/` |
| 多步骤+分支+边界处理 | `skills/` |
| 外部工具推荐/配置 | `tools/` |
| 感悟/思考/经验总结 | `articles/` |

> 仅当用户说「不确定」时才执行分类，并展示推荐让用户确认。用户已明确类型则跳过。

**等待用户确认类型和名称后进入 Step 2。**

---

## Step 2：设计资产结构

根据资产类型，确定内容骨架。不生成内容，只列出结构大纲让用户确认。

### 技能（skills/）结构

```
[SKILL.md]
├── frontmatter（name + description，含触发词）
├── 概述（一句话 + 核心原则）
├── Step 1 ~ N（每个 Step 含 I/O + 分支）
├── 边界情况（独立章节）
├── 禁止清单
└── 版本记录
```

### 命令（commands/）结构

```
[命令名].md
├── 名称 + 简介
├── 使用方式（完整的命令文本）
├── 适用场景
└── 注意事项
```

### 提示词（prompts/）结构

```
[名称].md
├── 名称 + 用途
├── 模板（含 {{占位符}}）
├── 示例（输入 → 输出）
└── 适用场景
```

### 角色（roles/）结构

```
[角色名].md
├── 名称 + 定位
├── 角色设定（身份/背景/能力）
├── 行为准则
├── 禁止行为
└── 输出风格
```

### 规则（rules/）结构

```
[规则名].md
├── 名称 + 作用
├── 规则内容（以「应当」「不得」表述）
├── 适用范围
└── 优先级
```

### 脚本（scripts/）结构

```
[脚本名].[ext]
├── 依赖
├── 使用方式
├── 参数说明（表格）
└── 示例
```

### 工具（tools/）结构

```
[工具名].md
├── 名称 + 简介
├── 推荐理由
├── 安装/配置
├── 使用方式
└── 注意事项
```

### 文章（articles/）结构

```
[标题].md
├── 标题 + 概述
├── 背景
├── 过程（话题板块）
├── 关键决策
├── 产出清单
└── 思考与备注
```

### 2.1 展示结构确认

```
📐 资产结构：[类型] — [名称]

[结构大纲]

确认这个结构？（Y / 调整 / 取消）
```

**等待用户回复：**
- Y → 进入 Step 3
- 调整意见 → 按意见修改后再次确认（最多 3 轮，第 3 轮后直接使用当前结构）
- 取消 → 流程终止

## Step 3：生成内容（内置质量检查）

按确认的结构逐节生成内容。**生成时自动执行以下质量规则，不询问用户。**

### 3.1 技能类型专属质量规则（强制执行）

> 以下规则来自 git-commit、session-summary、idea-classifier 三次审计的经验总结。

| # | 规则 | 要求 |
|---|------|------|
| **R1** | 触发词充足 | description 字段枚举 ≥3 个触发词，覆盖正式说法和口语说法，以「当用户说 X 时触发」收尾 |
| **R2** | 步骤 I/O 显式 | 每个 Step 结束时明确：继续下一步 / 等待用户确认 / 输出某格式后结束 |
| **R3** | 等待多路径 | 每个等待步骤至少提供 Y / 修改 / 取消 三条路径 |
| **R4** | 循环有退出 | 任何编辑-确认、补充-重分析循环都有显式上限：`最多 N 轮，第 N 轮后直接 [处理方式]` |
| **R5** | 边界有章节 | 独立的「边界情况」章节，覆盖典型异常（取消、缺失、冲突、过长/过短等） |
| **R6** | 输出有模板 | 关键输出用固定格式（代码块模板、表格），有禁止清单 |
| **R7** | 版本有记录 | 末尾版本记录表 |

### 3.2 非技能类型质量规则（降级版）

| 类型 | 必须满足 |
|------|----------|
| 命令 | 完整命令文本 + 适用场景 + 注意事项 |
| 提示词 | 占位符标记 `{{}}` + 至少一个输入输出示例 |
| 角色 | 角色设定 + 行为准则 + 禁止行为 |
| 规则 | 以「应当」「不得」句式表述 + 适用范围 |
| 脚本 | 依赖 + 执行命令 + 参数表 + 示例 |
| 工具 | 推荐理由 + 安装配置 + 基本用法 |
| 文章 | 模板章节完整（背景/过程/决策/产出） |

### 3.3 展示预览

生成后展示完整内容：

```
📄 资产预览：[类型] — [名称]

[完整内容]

确认写入？（Y / 修改 / 取消）
```

**等待用户回复：**
- Y → 进入 Step 4
- 修改意见 → 修改后再次预览（最多 3 轮，第 3 轮后直接询问 Y/N）
- 取消 → 流程终止

---

## Step 4：写入仓库

### 4.1 确定文件和路径

```
[项目根目录]/[类型目录]/[文件名]
```

命名规则：
- 技能：目录用 kebab-case（如 `my-new-skill/`），文件固定 `SKILL.md`
- 命令/提示词/角色/规则/文章：语义化 kebab-case（如 `weekly-report-template.md`）
- 脚本：`[功能].[扩展名]`（如 `batch-translate.py`）
- 工具：`[工具名].md`

若文件已存在，询问：覆盖 / 追加编号 / 取消。

### 4.2 写入文件

用 `write` 工具写入。

### 4.3 更新索引

若对应目录有 `INDEX.md`，追加新条目。

### 4.4 输出结果

```
✅ 资产已创建！

📁 路径：`[目录]/[文件名]`
🏷️ 类型：[类型]
📝 名称：[名称]
✅ 质量检查：[列出通过的规则，如 R1-R7 全部通过]

后续使用：[一句话指引]
```

流程结束。

## 边界情况

| 情况 | 处理 |
|------|------|
| 用户想基于现有资产改版 | 读取原文件，在 Step 2 展示差异结构，按变更部分生成 |
| 用户要创建的技能和已有技能高度重叠 | 提醒用户并建议：A. 合并到已有技能 / B. 作为已有技能的变体 / C. 继续独立创建 |
| 目录下 INDEX.md 不存在 | 按该目录的 INDEX.md 模板创建（参考已有 INDEX.md 格式） |
| 用户提供一个已有的 SKILL.md 要求优化 | 进入优化模式：直接执行六维度审计，修复发现的 ❌ 和 ⚠️ 条目 |
| 用户中途取消 | 任何等待环节用户说「算了/取消」→ 立即终止 |
| 用户在非 pi 环境使用 | 仅执行 Step 1-3，Step 4 提示手动保存 |

## 禁止清单

- ❌ 禁止生成没有触发词的技能 description（违反 R1）
- ❌ 禁止生成没有退出条件的循环结构（违反 R4）
- ❌ 禁止生成只有单一路径的等待步骤（违反 R3）
- ❌ 禁止跳过 Step 2 的结构确认直接生成内容
- ❌ 禁止跳过 Step 3 的预览确认直接写入文件
- ❌ 禁止生成空壳资产（只有骨架没有实质内容）
- ❌ 禁止在用户明确提供类型时仍执行分类判断

---

## 质量自检清单（生成后自动核对，不询问用户）

在 Step 3 生成内容后，内部执行以下核对：

```
[ ] R1 触发词 ≥3 个，覆盖正式+口语
[ ] R2 每步有 I/O 衔接
[ ] R3 每个等待有多路径
[ ] R4 每个循环有退出上限
[ ] R5 边界情况独立章节
[ ] R6 输出模板 + 禁止清单
[ ] R7 版本记录表
```

全部通过后才展示预览。若某项未通过，自行修复后重新核对。

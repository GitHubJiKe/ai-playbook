# Prompts Audit — 提示词质量审计

**核心定位：一个好的 Prompt 是可复用的输入模板——任何人拿到它，填入自己的变量，就能稳定地得到预期的输出。**

本 Skill 对照六个质量维度，检查一个提示词文件是否达到了「即拿即用」的标准——即：一个新用户打开这个文件后，不需要额外询问，就能理解它的用途、填入正确的参数、并预期会得到什么样的输出结果。

---

## 六个审计维度（Prompt 通用）

> **维度分级依据（标准层 vs 实践层）**
>
> P1-P2 是 🔴 高风险维度，对应**标准层（Hard Constraint）**：违反会导致用户「不知道这个 Prompt 干什么用」或「不知道怎么填入参数」，Prompt 基本等于不可用。
>
> P3-P6 是 🟡🟢 中低风险维度，对应**实践层（Soft Constraint）**：缺失会降低输出质量和一致性，但不影响 Prompt 的基本可用性。
>
> 审计时若遇到边界情况，用这个问题判断归属：「这个问题的缺失，是否导致一个新用户完全无法使用这个 Prompt？」→ 是 → 标准层（🔴）；「用户能用但效果不稳定？」→ 是 → 实践层（🟡🟢）。

| #   | 维度               | 核心问题                                                     | 层级      |
| --- | ------------------ | ------------------------------------------------------------ | --------- |
| P1  | **目标清晰度**     | 用户读完第一段就知道这个 Prompt 输入什么、产出什么吗？       | 🔴 标准层 |
| P2  | **变量设计**       | 所有占位符是否标记清晰、有填写说明和示例值？必填/可选是否区分？ | 🔴 标准层 |
| P3  | **输出格式锁定**   | 预期输出是否有明确的格式模板？有没有正例和反例？             | 🟡 实践层 |
| P4  | **防护与边界**     | 是否预防了已知的 failure mode？空输入、异常输入有没有处理？  | 🟡 实践层 |
| P5  | **示例完整性**     | 是否有至少一个完整的端到端示例？示例覆盖主场景了吗？         | 🟢 实践层 |
| P6  | **平台与版本**     | 是否标注了适用平台/模型？是否有版本记录？                    | 🟢 实践层 |

---

## 工作流

### Step 1：读取目标 Prompt

接收用户提供的提示词文件（文件路径 / 粘贴内容 / 文件名均可）。

- 若用户提供的是路径，用 `read` 读取
- 若用户直接粘贴内容，直接进入 Step 2
- 若用户只说了文件名（如「审计 九宫格卡通头像生成」），在 `prompts/` 目录下查找

读取成功后，**不输出任何内容**，直接进入 Step 2。

---

### Step 2：逐维度审计

对每个维度，按以下逻辑判定：

#### P1 — 目标清晰度

检查文件的**概述/标题区域**（通常是文件开头的第一段或「概述」章节）：

- ✅ **通过**：第一段/概述中明确说明了：① 这个 Prompt 的用途（干什么） ② 用户需要提供什么输入 ③ 预期产出什么输出。读完三秒内就能判断「这个 Prompt 是不是我需要的」。
- ⚠️ **警告**：说明了用途但「输入→输出」链条不完整——只说了做什么，没说要给什么；或标题模糊（如「图片生成」而非「九宫格卡通头像生成」）。
- ❌ **缺失**：没有概述段，直接跳入模板正文；或概述全是修饰词，没有实质信息。

> **特别注意**：对于多平台 Prompt 文件（如同时含 DALL·E / Midjourney / SD 的提示词），检查是否在每个子模板前标注了适用平台。

#### P2 — 变量设计

逐个检查文件中所有 `{{占位符}}` 或 `[变量]` 或其他需要用户替换的部分：

- ✅ **通过**：每个变量都有：① 明确的占位符标记（如 `{{风格关键词}}`） ② 填写说明（告诉用户这个位置填什么） ③ 至少一个示例值（如 `{{风格关键词}}` → 示例：`日系动漫 / 美式迪士尼`）。同时，必填和可选变量有视觉区分（如标注 `（必填）` / `（可选）`）。
- ⚠️ **警告**：变量标记存在但不统一（有的用 `{{}}`，有的用 `[]`，有的用自然语言「在此处填入…」）；或部分变量缺少示例值。
- ❌ **缺失**：变量没有占位符标记，混在自然语言中（如「选择你喜欢的风格」而不是 `{{风格}}`）；或完全没有变量说明，用户不知道哪里需要自己填。

> **特别注意**：对于「上传照片」这类非文本变量（如图片输入），检查是否说明了格式要求（如「正面大头照」「JPG/PNG」「文件大小 < 5MB」等）。

#### P3 — 输出格式锁定

检查 Prompt 对**最终输出的格式约束**：

- ✅ **通过**：① 有明确的输出模板/结构描述（如 `3x3 grid, square format, nine headshots, clean background`） ② 有关键词清单告诉 AI 必须包含什么（如 `must include: different expressions, consistent character`） ③ 有负面约束（如 `no text, no watermark, no realistic photo style`）——三者具备。
- ⚠️ **警告**：只描述了输出「应该有什么」，但没有给强制性的格式指令（如「最好」「可以」「尽量」这类软约束）；或只有正面描述没有负面约束。
- ❌ **缺失**：输出格式完全开放，AI 可以自由发挥格式、数量、风格等关键属性。Prompt 本质上只是一个愿望描述而非指令。

> **判断诀窍**：把这个 Prompt 给三个不同的 AI，它们会产出格式一致的结果吗？如果大概率不一致 → ❌。

#### P4 — 防护与边界

检查 Prompt 是否**预防了已知的 failure mode**：

- ✅ **通过**：覆盖了以下至少 3 类防护：① 空输入/缺参数的兜底行为 ② 异常格式输入的拒绝/纠正提示 ③ 风格一致性约束（防止风格漂移） ④ 内容安全边界（防止不当输出） ⑤ 长度/数量上限控制。
- ⚠️ **警告**：只有 1-2 类防护，或防护描述模糊（如「注意质量」而非具体的质量检查点）。
- ❌ **缺失**：完全没有防护机制。用户给了异常输入，Prompt 没有告诉 AI 该怎么应对。

> **特别注意**：对于生图类 Prompt，检查是否有风格一致性约束（防止九宫格中某个格子画风突变）和内容安全边界（防止生成不当内容）。

#### P5 — 示例完整性

检查 Prompt 文件中是否有**完整的使用示例**：

- ✅ **通过**：① 至少有一个完整的端到端示例（从用户输入/参数填入 → 到预期的输出描述或实际输出） ② 示例覆盖了最典型的使用场景 ③ 如果是多风格/多场景 Prompt，每个主要变体都有示例或至少有一个代表性示例。
- ⚠️ **警告**：有示例但不完整（只有输入没有输出，或输出只是文字描述不是实际效果）；或示例只覆盖了部分场景。
- ❌ **缺失**：完全没有使用示例。用户只能靠想象来推测效果。

> **判断诀窍**：一个新用户只看示例，不看模板正文，能成功使用吗？如果能 → ✅。

#### P6 — 平台与版本

检查 Prompt 文件的**元信息完整性**：

- ✅ **通过**：① 标注了适用的 AI 平台/模型（如 `DALL·E 3 / GPT-4o / Midjourney v6 / SDXL`） ② 不同平台的差异有说明（如「Midjourney 需加 `--ar 1:1`」） ③ 文件末尾有版本记录表（日期 + 变更内容）。
- ⚠️ **警告**：有平台标注但不完整（如只写了「AI 生图」没写具体平台）；或有多平台内容但没说各平台的注意事项；或有版本记录但缺少日期。
- ❌ **缺失**：完全没有平台信息和版本记录。用户不知道这个 Prompt 在哪个平台用，也不知道它是不是最新版。

### Step 3：输出审计报告

报告格式：

```
## Prompts Audit 报告：{{Prompt 名称}}

---

### 审计结果

| 维度 | 评级 | 说明 |
|------|------|------|
| P1 目标清晰度 | ✅/⚠️/❌ | [一句话说明原因，定位到具体位置] |
| P2 变量设计 | ✅/⚠️/❌ | [指出哪个变量有问题，缺什么] |
| P3 输出格式锁定 | ✅/⚠️/❌ | [指出缺了输出模板/正例/反例/禁止项中的哪一项] |
| P4 防护与边界 | ✅/⚠️/❌ | [覆盖了哪些防护，缺了哪些] |
| P5 示例完整性 | ✅/⚠️/❌ | [有无示例，是否完整覆盖场景] |
| P6 平台与版本 | ✅/⚠️/❌ | [平台标注是否完整，版本记录是否规范] |

---

### 问题清单（按优先级）

🔴 必须修复（❌ 条目）

问题：[具体描述，定位到文档的哪个章节/段落]
影响：[这个缺陷会导致什么实际问题]
修复建议：[给出具体应该怎么写，可以直接复制的内容优先]

🟡 建议改进（⚠️ 条目）

问题：[具体描述]
影响：[影响程度]
修复建议：[具体改进方案]

---

### 综合评级

基于 P1-P6 六个维度：

🏆 **优秀**（6/6 通过）：这个 Prompt 设计专业，任何人都可以即拿即用。
✅ **达标**（≥4 通过，无 ❌）：基本可用，建议修复 ⚠️ 问题后更稳定。
⚠️ **待改进**（有 1-2 个 ❌）：存在影响可用性或一致性的缺陷，建议修复后再推广。
❌ **不达标**（≥3 个 ❌）：用户拿到手大概率不知道如何使用或产出一致的结果，需要重构。

---

请选择下一步：
- 回复「修复」：我逐条帮你修改文件
- 回复「只修复 🔴」：只修复必须修复的条目
- 回复「结束」：查看报告后自行处理
```

---

### Step 4：执行修复（可选）

用户回复「修复」或「只修复 🔴」后：

1. 按 🔴 → 🟡 顺序，逐条处理
2. 每条修复展示「修改前 vs 修改后」的 diff，不整节重写
3. 修复完成后，**自动重新执行 Step 2**，验证修复是否生效
4. 若修复引入了新问题（如补了变量说明但引入了歧义），告知用户并继续修复

**退出条件：** 无 🔴 条目时输出 `✅ 必须修复项已清零，Prompt 达到最低可用标准`，或用户随时回复「结束」。

## 边界情况

### 用户提供的是一个极简 Prompt（< 100 字）

短 Prompt 不一定差——有些 Prompt 追求极简。审计时注意：

- P5（示例完整性）和 P4（防护与边界）对极简 Prompt 可适当放宽
- 重点关注 P1（目标清晰度）和 P2（变量设计），这两个维度对任何 Prompt 都关键
- 如果极简 Prompt 没有变量，P2 标记 `N/A（无变量结构）`，不计入失分

### 用户提供的 Prompt 是英文写的

按同样的六个维度审计，所有输出用中文，但引用 Prompt 原文时保留英文。

### 用户想对比两个 Prompt 的效果

分别执行两次审计，最后输出对比表：

```
| 维度 | Prompt A | Prompt B |
|------|----------|----------|
| P1   | ✅       | ⚠️       |
| ...  | ...      | ...      |
```

并给出「推荐使用哪个版本 / 如何融合两者优点」的建议。

### 用户说「这个 Prompt 效果不好，帮我找原因」

不要只做结构审计——在报告末尾增加「执行稳定性推断」章节：

根据审计结果，推断 AI 执行这个 Prompt 时最可能出现的偏差（如「P3 缺失导致 AI 可能在输出中混合多种风格，无法保证九宫格的一致性」「P4 缺失导致用户上传侧脸照片时 AI 不会主动提醒」）。

### 用户提供的是系统提示词（角色设定类 Prompt）

系统提示词比普通 Prompt 多了角色设定和工作流。除了 P1-P6 外，追加检查：

- 角色设定是否有「禁止行为」清单
- 工作流是否有明确的步骤边界
- 是否区分了「内部指令」和「对用户说的话」

在报告的「审计结果」表格后追加一段「系统提示词附加检查」，格式：

```
### 系统提示词附加检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 角色边界清晰 | ✅/⚠️ | [说明] |
| 工作流完整 | ✅/⚠️ | [说明] |
| 内外部语言区分 | ✅/⚠️ | [说明] |
```

## 输出风格要求

- 问题描述要**定位到具体位置**：指出是「概述段第二句」「变量 `{{风格}}` 的说明」「P3 输出格式章节」，不说「不够清晰」这种废话
- 修复建议要**可以直接复制**：给出具体的文字片段，而不是「建议补充示例」
- 评级要**诚实**：有 ❌ 就标 ❌，不因为「Prompt 写得长」就升级
- P2 的 N/A 要**明确说明理由**：「此 Prompt 无占位变量，该维度不适用」

---

## 禁止清单

- ❌ 禁止在不读取 Prompt 内容的情况下直接给出审计结果
- ❌ 禁止跳过 Step 3 的报告格式，随意输出审计意见
- ❌ 禁止把「篇幅长」等同于「质量高」——短 Prompt 也可能所有维度通过
- ❌ 禁止在用户表达不确定时强行给结论——可追问用户澄清
- ❌ 禁止在修复时整节重写而非精准 diff——最小改动原则

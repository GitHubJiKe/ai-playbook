# Idea Classifier — 想法分类与沉淀

帮用户将一个模糊的想法/需求/点子，判断它在 AI Playbook 仓库中最合适的资产类型，然后按该类型的格式生成内容并写入对应目录。

> ⚠️ **核心原则**：分类有据，输出有格。每个分类推荐必须给出明确的「为什么是 A 而不是 B」的推理。生成的资产必须符合目标目录的格式规范，不能只是一个草稿。

---

## 八种资产类型定义与判定标准

| 类型 | 目录 | 本质定义 | 判定关键词 |
|------|------|----------|-----------|
| **命令** | `commands/` | 一句话指令，单轮对话即可完成，无状态 | 「给我…」「帮我…」一次搞定 |
| **提示词** | `prompts/` | 可复用的模板，有占位变量，聚焦输出格式 | 「每次…都这样问」「模板」「格式」 |
| **角色** | `roles/` | AI 的人设定义，规定 AI 是谁、有什么专业背景、行为边界 | 「扮演…」「你是…」「从…的视角」 |
| **规则** | `rules/` | 跨场景的行为约束，附加到其他 prompt 上生效 | 「始终…」「永远不要…」「注意…」 |
| **脚本** | `scripts/` | 可执行的自动化代码，在聊天之外运行 | 「批量」「自动化」「定时」「流程」 |
| **技能** | `skills/` | 多步骤完整工作流，含分支、边界处理、输入输出 | 「先…再…然后…」「如果…则…」分步 |
| **工具** | `tools/` | 外部工具配置、推荐清单、使用指南 | 「用什么工具」「配置」「推荐」 |
| **文章** | `articles/` | 非可执行资产：感悟、思考、经验总结 | 「我觉得…」「经验」「思考」「感想」 |

---

## Step 1：收集用户想法

用户触发技能后，首先确认要沉淀的内容：

### 1.1 如果用户已描述清楚

直接进入 Step 2。

### 1.2 如果用户只说了触发词但没描述想法

询问：

```
好的，请描述你的想法/需求：你想解决什么问题？期望达到什么效果？有什么具体的使用场景吗？
```

### 1.3 引导补充（若信息不足）

从以下三个角度追问，直到足以做出分类判断：

1. **使用频率**：是一次性的还是反复使用？
2. **复杂度**：一句话能说清，还是需要分步骤？
3. **聊天内/外**：在 AI 对话中使用，还是需要写代码自动化？

> 最多追问 2 轮，若仍无法判断，给出「暂无法分类」建议并让用户自行决定。

---

## Step 2：分类分析

根据收集到的信息，对照八种类型的判定标准，进行一次**内部分析（不输出给用户）**：

### 2.1 排除法初筛

快速排除明显不匹配的类型：

- 只是感想/反思 → 排除命令/提示词/脚本等可执行类型，聚焦 `article`
- 需要写代码运行 → 排除角色/规则等，聚焦 `script`
- 一句话指令 → 排除技能/角色，聚焦 `command` 或 `prompt`

### 2.2 正选法确认

对剩余候选类型，回答：

- 这个想法**最像**哪种类型的典型用例？
- 如果用最不合适的类型去装，会出现什么问题？（反向验证）

### 2.3 边界类型判断

当想法落在两个类型的边界时，按以下规则判定：

| 边界场景 | 判定方向 | 理由 |
|----------|----------|------|
| 命令 vs 提示词 | 如需占位变量或格式化输出 → 提示词；纯指令 → 命令 | 提示词的核心是有模板骨架 |
| 提示词 vs 角色 | 如要求 AI 以特定身份/视角系统性地思考 → 角色；仅控制输出格式 → 提示词 | 角色定义的是「谁在说话」 |
| 角色 vs 规则 | 如定义完整的 AI 人格 → 角色；只是一条行为约束 → 规则 | 角色是 Who，规则是 How |
| 规则 vs 命令 | 附加性的、跨场景 → 规则；独立完成一个任务 → 命令 | 规则不独立完成任务 |
| 脚本 vs 技能 | 在聊天外用代码运行 → 脚本；在聊天内多步交互 → 技能 | 边界是运行环境 |
| 技能 vs 命令 | 需要分步 + 分支 + 边界处理 → 技能；单轮即可 → 命令 | 复杂度分水岭 |
| 工具 vs 脚本 | 推荐/配置外部工具 → 工具；自动化执行 → 脚本 | 工具是「用什么」，脚本是「怎么做」 |

## Step 3：展示分类结果

输出分类推荐，包含推理过程：

```
## 分类建议

**推荐类型**：[类型名] → `[目录]/`

**推荐理由**：
- [理由 1：为什么是这个类型]
- [理由 2：为什么不是其他容易混淆的类型]

**备选类型**（如有）：[备选类型]，适用场景：[什么情况下备选更合适]

**资产定位**：
- 存放路径：`[目录]/[文件名]`
- 使用方式：[给用户的一句话说明，这个资产之后怎么用]

确认这个分类？（Y / 调整类型 / 补充信息）
```

**等待用户回复：**
- Y → 进入 Step 4
- 用户指定其他类型 → 以用户选择为准，进入 Step 4（但可简短提醒为什么原推荐更合适）
- 用户补充信息 → 回到 Step 2 重新分析（最多 2 轮，第 2 轮后直接给出分类推荐，不再接受补充）

---

## Step 4：生成资产内容

根据确认的类型，按对应格式生成内容。

### 4.1 各类型输出格式

#### 命令（commands/）

```markdown
# [命令名称]

一句话说明这个命令做什么。

## 使用方式

直接对 AI 说：

> [完整的命令文本]

## 适用场景

- [场景 1]
- [场景 2]

## 注意事项

- [注意事项]
```

#### 提示词（prompts/）

```markdown
# [提示词名称]

一句话说明这个提示词的用途。

## 提示词模板

[完整的提示词文本，用 `{{占位符}}` 标记需要替换的部分]

## 使用示例

**输入**：[示例输入]
**输出**：[期望得到的输出]

## 适用场景

- [场景]
```

#### 角色（roles/）

```markdown
# [角色名称]

角色定位的一句话描述。

## 角色设定

你是 [角色描述：身份、专业背景、核心能力]。

## 行为准则

- [准则 1]
- [准则 2]

## 禁止行为

- [禁止 1]
- [禁止 2]

## 输出风格

[风格描述]
```

#### 规则（rules/）

```markdown
# [规则名称]

一句话说明这个规则的作用。

## 规则内容

[规则的具体内容，以「你应当…」「你不得…」句式表述]

## 适用范围

[哪些场景下应用此规则]

## 优先级

[与其他规则冲突时的优先级说明]
```

#### 脚本（scripts/）

```markdown
# [脚本名称]

一句话说明脚本的功能。

## 依赖

- [依赖项]

## 使用方式

[执行命令或调用方式]

## 参数说明

| 参数 | 说明 | 默认值 |
|------|------|--------|
| [参数] | [说明] | [默认值] |

## 示例

[示例输入输出]
```

#### 技能（skills/）

按标准 SKILL.md 格式生成（参考已有技能的结构：frontmatter → 概述 → 步骤 → 边界情况 → 禁止清单 → 版本记录）。

#### 工具（tools/）

```markdown
# [工具名称]

一句话说明这个工具。

## 推荐理由

[为什么推荐]

## 安装/配置

[如何安装或配置]

## 使用方式

[基本用法]

## 注意事项

- [注意点]
```

#### 文章（articles/）

按 session-summary 技能的文章模板生成（标题 → 概述 → 正文）。

### 4.2 展示预览

生成后展示给用户：

```
📄 资产预览：[类型] — [名称]

[完整内容预览]

确认写入？（Y / 修改 / 取消）
```

**等待用户回复：**
- Y → 进入 Step 5
- 用户提供修改意见 → 修改后再次预览（最多 3 轮，第 3 轮后直接询问 Y/N）
- N / 取消 → 流程终止

---

## Step 5：写入仓库

### 5.1 确定文件路径

```
[项目根目录]/[目录]/[文件名]
```

文件命名规则：
- 命令/提示词/角色/规则/文章：语义化中文名或英文 kebab-case
- 脚本：`[功能名].[扩展名]`（如 `.py`、`.sh`）
- 技能：目录名为 kebab-case，文件固定为 `SKILL.md`

### 5.2 写入文件

用 `write` 工具将内容写入对应路径。

### 5.3 更新索引

若对应目录存在 `INDEX.md`（`skills/`、`tools/` 等），在索引文件中追加新条目。

### 5.4 输出结果

```
✅ 已归档！

📁 路径：`[目录]/[文件名]`
🏷️ 类型：[类型]
📝 名称：[名称]

后续使用：[一句话指引]
```

流程结束。

## 边界情况

| 情况 | 处理 |
|------|------|
| 用户的想法横跨多种类型 | 优先推荐最核心的类型；若确实需要拆分，建议「先做一个，后续再补其他」 |
| 用户坚持一个明显不合适的分类 | 简短提醒一次原因（如「这个想法需要分步骤和分支处理，放在 commands/ 里会丢失大量结构」），若用户仍坚持则尊重选择 |
| 想法太模糊无法分类 | 追问 2 轮后仍无法判断 → 给出「建议先以文章形式记录想法，后续明确后再转化」的方案 |
| 文件已存在 | 询问：覆盖 / 追加编号后缀 / 取消 |
| 用户中途取消 | 任何等待环节用户说「算了/取消」→ 立即终止，不写入任何文件 |
| 用户反复补充/修改 | 分类环节最多 2 轮补充，生成环节最多 3 轮修改，之后仅提供 Y/N 或直接输出 |
| 用户在非 pi 环境使用本技能 | 仅执行 Step 1-4（分类 + 生成内容），Step 5 提示用户手动保存到对应目录 |

---

## 禁止清单

- ❌ 禁止在没有完整分类推理的情况下直接给出分类结果
- ❌ 禁止跳过 Step 3 的用户确认，直接生成和写入
- ❌ 禁止将技能分类为命令（丢失结构化流程）或将命令分类为技能（过度设计）
- ❌ 禁止在用户表达不确定时强行推荐一个类型——可给「暂无法判断」结论
- ❌ 禁止生成空壳资产（只有标题没有实质内容）

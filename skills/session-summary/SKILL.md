# Session Summary

将当前与 AI 的整个对话上下文提炼为一篇结构清晰的文章，记录这次协作的全过程——做了什么、为什么这样做、关键决策、产出结果，保存到 `articles/` 目录。

> ⚠️ **核心原则**：服务于读者，而非归档。文章的目标读者是未来的自己或其他人，他们需要快速理解这次对话做了什么、为什么、有什么值得记住的思考。不是流水账，是有脉络的叙事。

---

## Step 1：扫描对话上下文

回顾当前对话的全部消息（用户和 AI 的所有轮次），在内部完成以下提取，**不向用户输出**：

### 1.1 主题识别

- 这次对话围绕什么核心目标展开？（1 句话）
- 有几个主要话题板块？（通常 2-5 个）

### 1.2 关键节点提取

对每个话题板块，提取：

| 维度 | 说明 |
|------|------|
| 触发 | 用户说了什么启动了这个话题 |
| 过程 | 经过了哪些关键交互（确认、修正、迭代） |
| 决策 | 做了哪些选择（如风格选 C、主题选跟随系统） |
| 产出 | 创建/修改了哪些文件，做了什么操作 |
| 转折 | 有没有方向调整或问题修复 |

### 1.3 元信息提取

- 对话大致时长/日期
- 涉及的核心仓库/项目
- 涉及的技能（如有）

---

## Step 2：确定标题和文件名

### 2.1 生成标题

根据 Step 1 的主题识别，生成一个有意义的文章标题。标题应：
- 概括这次对话的核心主题
- 可读性强，不是技术日志风格
- 示例好标题：「AI Playbook 仓库初始化与技能搭建」「为 ai-playbook 搭建 Markdown 转 HTML 工作流」
- 示例差标题：「2026-06-13 对话记录」「与 AI 的一次对话」

### 2.2 确定文件名

两种策略，按需选择：

**策略 A：语义化命名（优先）**

取标题的英文/拼音简化版，kebab-case 格式：
- 「AI Playbook 仓库初始化」→ `ai-playbook-init.md`
- 「搭建 MD to HTML 技能」→ `md-to-html-skill-setup.md`

**策略 B：时间戳命名（兜底）**

当语义难以概括（对话主题分散、难以一句话总结）时使用：
- 格式：`YYYYMMDD-HHMMSS.md`
- 示例：`20260613-143025.md`

> 默认优先使用策略 A。如果标题难以转换为有意义的文件名（过于泛化、多主题交织），降级到策略 B。

### 2.3 展示给用户确认

```
📝 文章标题：「[标题]」
📄 文件名：`[文件名]`
📁 保存路径：`articles/[文件名]`

确认？（Y / N / 修改标题）
```

**等待用户回复：**
- Y → 进入 Step 3
- N → 询问用户想要的标题，重新生成（最多 3 轮，第 3 轮后直接使用当前标题进入 Step 3）
- 用户提供修改意见 → 按意见调整后再次确认（最多 3 轮，第 3 轮后直接使用当前标题）

> 用户说「直接写/不用确认」→ 跳过确认，使用生成的标题和文件名，直接进入 Step 3。

## Step 3：撰写文章

按以下模板结构撰写文章，以 Markdown 格式输出到 `articles/` 目录。

### 文章模板

```markdown
# [标题]

> [一句话概述这次对话做了什么]

## 背景

[1-2 段：为什么有这次对话？出发点和上下文是什么？]

## 过程

### [话题板块 1]

[做了什么，关键决策和原因，产出。不需要逐句还原对话，概括脉络即可。]

### [话题板块 2]

[同上]

（话题板块一般 2-5 个）

## 关键决策

| 决策 | 原因 |
|------|------|
| [决策 1] | [为什么] |
| [决策 2] | [为什么] |

## 产出清单

| 文件/操作 | 说明 |
|-----------|------|
| [路径 1] | [做了什么] |
| [路径 2] | [做了什么] |

## 思考与备注

[可选：这次对话中值得记录的思考、踩坑、收获，1-3 点即可]
```

### 撰写原则

- **高度概括**：不逐句还原对话，用 2-5 句话概括一个话题板块
- **面向读者**：让没参与对话的人也能看懂脉络
- **细节用量**：关键决策保留细节（如为什么选风格 C），执行细节可省略（如具体代码片段不用贴）
- **篇幅控制**：整篇文章控制在 500-1500 字，不追求完整转录
- **代码和文件**：产出清单列出文件路径即可，不贴文件内容
- **禁止**：不要输出对话的逐字记录，不要贴完整的 AI 回复内容

---

## Step 4：写入文件

### 4.1 检查目录

确认 `articles/` 目录存在：
```bash
ls [项目根目录]/articles/
```

若不存在，创建：
```bash
mkdir -p [项目根目录]/articles/
```

### 4.2 写入文件

用 `write` 工具将撰写好的 Markdown 文章写入：

```
[项目根目录]/articles/[文件名]
```

### 4.3 更新索引

在 `articles/INDEX.md` 中追加新条目。若 INDEX.md 不存在，先按模板创建。

索引表格追加一行：

```
| [序号] | [标题] | [文件名] | ✅ 已完成 | [一句话简介] |
```

### 4.4 输出结果

```
✅ 文章已保存！

📄 文件：`articles/[文件名]`
📝 标题：「[标题]」
📏 篇幅：约 [字数] 字

可在 GitHub 上查看或通过浏览器打开阅读。
```

流程结束。

## 边界情况

| 情况 | 处理 |
|------|------|
| 对话非常短（< 5 轮） | 提示：「本次对话较短（仅 [N] 轮），是否仍然生成文章？」→ 等用户确认 |
| 对话非常长（> 50 轮） | 只提取核心脉络，不追求完整覆盖。在文章开头注明：「本次对话较长，以下为关键脉络摘要」 |
| 对话主题过于分散 | 标题使用「[日期] 杂记」类 + 时间戳文件名（策略 B），在文章中按时间线简述 |
| 对话中未产生文件产出 | 产出清单改为「本次对话以讨论为主，无文件产出」，保留决策表 |
| 用户在某一步骤说「算了/取消」 | 立即终止，不写入任何文件 |
| 同名文件已存在 | 文件名追加 `-2`、`-3` 后缀，或改用时间戳命名 |
| 用户反复修改标题 | 最多允许 3 轮调整，之后直接使用当前标题 |
| 用户说「只写某个话题」 | 只提取该话题的内容，标题和文件命名聚焦该话题 |

---

## 禁止清单

- ❌ 禁止逐轮还原对话（如「用户说：... AI 说：...」）——这是文章，不是聊天记录
- ❌ 禁止贴代码片段或文件完整内容——引用路径即可
- ❌ 禁止在未确认标题的情况下直接写入文件（除非用户说跳过确认）
- ❌ 禁止将对话中所有细节无差别记录——保持概括和筛选
- ❌ 禁止用「首先...其次...然后...最后...」的流水账模式——用话题板块组织

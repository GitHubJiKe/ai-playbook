# 从三次审计到一套流水线：AI Playbook 资产创建体系的建立

> 一次技能审计马拉松，发现了三个技能共有的同一个缺陷。这个发现催生了一套完整的资产创建流水线——从分类、创建到质检，全链条自动化。

## 背景

AI Playbook 仓库已经有了四个技能（git-commit、session-summary、md-to-html、skills-audit），但它们的质量如何？有没有什么隐藏的结构性问题？决定用 skills-audit 技能对它们做一次全面审计，于是有了后面一连串的发现和建设。

## 过程

### 审计三部曲：同一个缺陷，三次出现

首先审计 **git-commit**。六维度中五个全绿，唯独 D4（退出条件）亮了黄灯——Step 3.3 的 commit message 编辑循环没有最大轮次限制，理论上用户可以无限修改。修复很简单：在循环路径上加一句「最多 3 轮」。修复后 6/6 🏆。

接着审计 **session-summary**。一模一样的剧本：D4 ⚠️，Step 2.3 的标题确认循环缺退出条件。同样的修复，同样的结果 🏆。

第三个是新建的 **idea-classifier**。这个技能是全新的——目的是帮用户判断一个想法该沉淀为哪种资产类型（命令、提示词、角色、规则、脚本、技能、工具、还是文章）。写完之后直接审计，发现 D4 又亮了黄灯——而且这次是**两处**循环都没加退出条件。修完三处后 🏆。

### 模式识别：这不是巧合

三个技能，同一个缺陷，不是巧合。这是系统性盲区：创建技能时，人的注意力在流程设计上，很容易忘记给循环加上限。

**决策点**：不能每次都事后修补，必须把这个教训固化为一个前置约束。

### 把教训变成规则

创建了一条规则文件 `skill-loop-exit-condition.md`，核心内容是四类循环模式的退出条件模板：

| 循环模式 | 退出条件 |
|----------|---------|
| 编辑-确认循环 | 最多 3 轮 → 直接询问 Y/N |
| 补充-重分析循环 | 最多 2 轮 → 直接给出推荐 |
| 修复-验证循环 | 无 🔴 时退出 |
| 等待-重试循环 | 最多 N 次 → 终止或降级 |

但一个问题浮现了：pi 的架构下，`rules/` 目录不会被自动加载到系统提示词中。规则文件静静地躺在目录里，pi 不知道它的存在。

### 从规则到技能：Skill Creator 的诞生

既然静态规则无法自动生效，那就把它嵌入到一个会动的技能里。

**Skill Creator** 被设计为资产创建器——不是只创建技能，而是覆盖全部八种资产类型。核心创新在于它内置了七条硬性质量规则（R1-R7），在 Step 3 生成内容时强制执行，不询问用户。其中 R4 就是「循环有退出」，直接杜绝了那三次审计中反复出现的缺陷。

更妙的是，Skill Creator 审计自己的结果：**首次生成即 6/6 🏆**。它践行了自己定下的规则。

### 完整流水线成形

至此，三条技能各司其职，形成了一条完整的资产创建流水线：

```
idea-classifier  →  判断该做成什么类型（八选一）
skill-creator    →  按类型生成高质量资产（内置 R1-R7 质检）
skills-audit     →  事后审计已有资产的质量（六维度评分）
```

## 关键决策

| 决策 | 原因 |
|------|------|
| D4 修复统一为「最多 N 轮」模式 | 三个技能都适用同一模式，保持一致性 |
| 先创建静态 rule 文件 | 作为仓库内可检索的知识沉淀 |
| 再将 rule 嵌入到 skill-creator 技能 | pi 不自动加载 rules，需要通过技能机制让约束自动生效 |
| Skill Creator 覆盖全部八种类型而非仅技能 | 最大化复用价值，不只解决技能场景 |
| 技能内置七条规则而非仅 R4 | 三次审计暴露的不止 D4，R1-R7 是全套防护 |

## 产出清单

| 文件/操作 | 说明 |
|-----------|------|
| `skills/git-commit/SKILL.md` | D4 修复：Step 3.3 编辑循环加退出上限 |
| `skills/session-summary/SKILL.md` | D4 修复：Step 2.3 标题确认循环加退出上限 |
| `skills/idea-classifier/SKILL.md` | 新建：八种资产类型的分类判定技能 |
| `skills/skill-creator/SKILL.md` | 新建：内置 R1-R7 质量规则的资产创建器 |

| `rules/INDEX.md` | 更新：补充新规则条目 |
| `skills/INDEX.md` | 更新：补充 idea-classifier 和 skill-creator 条目 |

## 思考与备注

1. **审计的价值不在发现问题，在发现模式。** 单个技能的 D4 缺失只是小修小补，但连续三个技能都犯同样的错误，说明这需要制度性解决而非个案修补。

2. **静态知识需动态载体。** Rule 文件写在 `rules/` 目录里只是存档，嵌入到 skill-creator 的执行流程里才是生效。在 AI Agent 的语境下，知识必须能"跑起来"才算真正可用。

3. **最好的质检是不需要事后审计。** Skill Creator 首次即 🏆 的事实证明：把质量规则前移到生成阶段，比事后修补高出不止一个档次。skills-audit 的终极目标，就是让自己变得越来越不需要。
